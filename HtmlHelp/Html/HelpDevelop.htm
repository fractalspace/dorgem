How to help develop Dorgem

If you want to help develop Dorgem there are various thing you can do:

Test the current version and report bugs you find
This is easy for everyone! If you find a bug, you can send it to me directly or enter them at the project space at SourceForge. Please report a bug as precisely as possible. If I can't reproduce it, I can fix it!

Suggest new features
New suggestions are always welcome, but please look at the To Do list first to see if they are not already on the list. You can send them to me directly or enter them at the project space at SourceForge.

Note: When you use SourceForge, please log in or leave an email address, so I can contact you.

Help coding new features
If you want to help coding you need to be familiar with MFC and have access to Visual C++ 6, since that is my development platform.

One of the reasons I put Dorgem up on SourceForge was so other developers could get involved. If you are seriously interested, first of all that a look at the code, try to build it and consider what you can add. After that please contact me to discuss what you want to do. This is to prevent two people develop the same thing, which is a waste of time for one of them.

Then grab a copy of the most recent sources from anonymous CVS, write your code, create a patch and send it to me. I'll merge it into the main codebase. If you never have used CVS before, take a look at TortoiseCVS. It integrates CVS with the Windows Explorer seamless.

If you do decide to join in, please follow these simple guidelines, which will ultimately make all our lives easier:

  • Do not make trivial changes to existing code, such as deleting blanks, consolidating multi-line comments into single line comments, etc. This just adds a bunch of needless differences to the code. Besides, if every developer massaged the code to fit his/her style, it would be a never-ending struggle.
  • Please try to match existing coding style as much as possible.
  • Don't use tabs but spaces, since everybody has different tab size. I use a tab size of 2.
  • Please make sure to comment your work so we can all understand what you've done (even though I didn't consistently ;-)
  • Always use single line comment (//) to place real comment.
  • Please don't make changes that don't relate to the task at hand. If you come across other issues while handling a task, please submit them as bugs.
  • Please don't make changes that aren't on the to-do list or bug list. If everyone just threw in everything they wanted, the product would become a hodge-podge.
  • You have the HTML Help Workshop installed in C:\Program Files\HTML Help Workshop. This because the full paths are stored in the VC project and MakeHelp.bat.

My general plan is to start people with anonymous CVS access and I commit their changes to the CVS repository myself. Those developers that continue to contribute will be considered for read-write CVS access.

Please note that in some source files special #define lines are used to (optionally) exclude certain parts of the code. These #define lines only affect the debug build. If you don't need, comment them out.
